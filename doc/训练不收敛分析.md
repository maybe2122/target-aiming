# 训练不收敛问题分析报告

> 生成时间：2026-04-19
> 分析范围：`source/target_aiming/target_aiming/tasks/direct/target_aiming/` 下的环境与 PPO 配置，以及 `logs/rsl_rl/target_aiming_direct/` 下最近几次 run。

---

## 结论先行

当前训练不收敛最可能是 **环境 + reward 设计问题**，而不是 PPO 超参。按严重程度排序的 4 个根因：

1. **动作速度过慢**：每步可旋转角度 ≪ reset 初始偏差，agent 物理上追不上目标。
2. **episode 过早终止**：目标一离 FOV 立即 done，配合边缘初始化，episode 普遍很短。
3. **reward 尺度失衡**：`idle` 惩罚比 `pixel` 主信号大一个数量级，pixel tracking 信号被淹没。
4. **reward 跳变尖峰**：visibility 翻转时 `pixel_error` 从 ~0.1 跳到 1.0，产生 ±9 的噪声，value 网络难以拟合。

其它次要问题（CNN 输入未归一化、batch 偏小、错配的 `init_noise_std`）会放大上述效应但不是主矛盾。

## 问题 1：动作速度过慢（最关键）

### 现象

```python
# target_aiming_env.py:51-59
step = self.cfg.fixed_step_rad          # 0.03
self._action_map = torch.tensor([
    [+step, 0.0], [-step, 0.0],
    [0.0, +step], [0.0, -step],
    [0.0,  0.0],
], device=self.device)

# target_aiming_env.py:112-122
def _apply_action(self) -> None:
    ...
    velocity_targets[:, idx] = self.actions[:, 0] * self.cfg.action_scale
    self.gimbal.set_joint_velocity_target(velocity_targets)
```

`_action_map` 里的 ±0.03 被作为 **velocity target (rad/s)** 传给 `set_joint_velocity_target`。

### 数量级分析

- 每个 policy step 实际旋转：`0.03 rad/s · 0.0333 s ≈ 0.001 rad ≈ 0.057°`
- 整集 10 s 的上限：`0.03 rad/s · 10 s = 0.3 rad ≈ 17°`
- 但 reset 时 yaw 随机偏差的一半 FOV 上限是 **24°**

也就是说：**一整集都未必能把 yaw 补回中心**。agent 学得再快，在物理层面也无法达成 `pixel_error < 0.10`（success_threshold）。

### 验证方式

跑 `rollout_diagnose.py --policy zero` 和 `--policy random`，看：
- `joint angles` 曲线每秒变化量 ≪ 1 rad → 确认
- `episode length histogram` 如果主要在 < 150 → 配合问题 2 也成立
- `pixel_error` 即使是 random 也几乎从不 < 0.10 → 进一步确认 success 几乎拿不到

### 修复建议

- 如果 `fixed_step_rad` 语义是 **"每个 policy step 目标角度增量"**，则 velocity 应该是 `fixed_step_rad / (decimation · sim.dt) ≈ 0.9 rad/s`。把 `action_scale` 改成 `1.0 / (decimation * sim.dt)` 即可，无需改 env 代码。
- 或者把 `_apply_action` 改成 **position control**：`joint_pos_target = cur + action_map[idx]`（需要 `set_joint_position_target`），同时把 `stiffness` 从 0 改到一个正值（否则 implicit actuator 没用）。

## 问题 2：出 FOV 立即终止

### 现象

```python
# target_aiming_env.py:138-142, 148-156
self._pixel_error_normalized = torch.where(
    visible, self._pixel_error_normalized, torch.ones_like(...),  # 不可见 → 强制 1.0
)
...
out_of_view = ~self._target_visible & (self._pixel_error_normalized > self.cfg.max_pixel_error)
```

因为不可见时 `pixel_error` 总是 1.0，`max_pixel_error=0.95` 永远满足，`out_of_view` 实际上退化成 `~visible`。

### 影响

- reset 时 yaw 偏差接近 24°（问题 1），目标容易在 FOV 边缘；动作稍有不对就被判 done，没有时间学习"先把目标拉回视野"。
- 大量 episode 长度远小于 300 step，PPO 的采样里成功轨迹比例极低。
- `Episode_Termination/out_of_view` 曲线通常会远高于 `Episode_Termination/time_out`。

### 修复建议

- 把 `max_pixel_error` 真正作为"最后的忍耐阈值"：让 `pixel_error` 在不可见时基于 3D 夹角之类连续增长（例如 `1 + 0.1·angle_error_rad`），然后 `out_of_view = pixel_error > 2.0`。
- 或者给 N 步宽限期：连续不可见超过 K=10 步才 done。
- 或者干脆只保留 `time_out`，不视觉上 terminate；用 `rew_alive` 的正/负号来激励保持可见。

## 问题 3：reward 尺度失衡

### 现象

```python
# target_aiming_env.py:336-352
error_delta = prev_pixel_error - pixel_error          # 每步 ~0.003 (问题 1 的直接后果)
rew_pixel   = 10.0 * error_delta                      # ≈ 0.03/步
rew_smooth  = -0.01 * sum(actions^2) ≈ -0.01 · 0.0009 ≈ -1e-5  # 可忽略
rew_success = 5.0 * (pixel_error < 0.10)              # 几乎拿不到(问题1)
rew_alive   = 0.1 * visible                           # ≈ 0.1/步(只要在视野内)
rew_idle    = -1.0 * is_idle * (pixel_error > 0.10)   # 惩罚静止
```

逐步量级：
| 分量 | 典型每步值 |
|---|---|
| pixel | +0.03 |
| smooth | ~0 |
| success | 0（追不上） |
| alive | +0.1 |
| idle | -1.0（只要选 stay） |

`idle` 的量级是 `pixel` 的 30+ 倍。policy 学到的第一件事会是"永远不选 stay"，而不是"朝目标中心对准"。

### 修复建议

- 先把 `rew_scale_idle` 调到 `-0.05 ~ 0`（或先设 0 看其他信号能不能学出来）。
- 把 `rew_scale_pixel_error` 调到 `50~100`，同时考虑加一个**绝对位置项**而不只是 delta：`-pixel_error * scale`，这样即使不动也有稳定的梯度方向。
- `rew_scale_success` 可以拉到 10 或更大，且可以把阈值放宽到 0.15/0.2 先让 agent 尝到甜头。

## 问题 4：reward 跳变尖峰

### 现象

当一步内 `visible` 从 True 翻到 False：
- `prev_pixel_error ≈ 0.1`（上一步还在视野里）
- `pixel_error = 1.0`（被强制置 1）
- `rew_pixel = 10 · (0.1 - 1.0) = -9`

reset 后下一步又反过来：
- `prev_pixel_error = 1.0`（reset 时遗留）
- `pixel_error ≈ 0.3`
- `rew_pixel = 10 · (1.0 - 0.3) = +7`

一条轨迹里会周期性出现 ±7~9 的尖峰，GAE 扩散到整个 episode。

### 影响

- `Loss/value_function` 居高不下；critic 把噪声拟合进去反而害 actor。
- PPO 自适应 lr 容易被触发降到 1e-5 以下，训练变慢。

### 修复建议

- reset 时 `_prev_pixel_error[env_ids] = self._pixel_error_normalized[env_ids]`（当前代码在 reset 里没覆盖）。
- 在 `compute_rewards` 里 **clip** `error_delta` 到 `[-0.1, 0.1]`，让 reward 分量的量级恒在合理范围。
- 或者使用 **绝对误差惩罚** `-pixel_error * scale` 替代 delta 形式，天然避免尖峰。

## 其它次要但值得一起改

### 5. CNN 观测未归一化、分辨率偏大
```python
# rsl_rl_ppo_cfg.py:33, 44
obs_normalization=False
# env_cfg
camera_width=256, camera_height=256
```
RGB 已经除以 255 在 `[0,1]`，但仍建议 `(x-0.5)/0.5` 或打开 `obs_normalization=True`（running mean/std）。
分辨率建议先降到 `128×128`，CNN 卷积层可以瘦身（kernel=[8,4,3], channel=[32,64,64]），训练速度和收敛都会更好。

### 6. PPO batch 对图像策略偏小
```
num_envs=64, num_steps_per_env=48, num_mini_batches=4
→ batch = 3072, minibatch = 768
```
图像 PPO 经验上 minibatch ≥ 4k 才稳；建议：
- `num_envs` 提到 256（如果显存允许）
- 或 `num_steps_per_env` 提到 128
- `num_mini_batches=8`

### 7. `init_noise_std=0.3` 对 Categorical 无效
```python
# rsl_rl_ppo_cfg.py:35-36
distribution_cfg=CategoricalDistributionCfg(),
init_noise_std=0.3,
```
`init_noise_std` 只作用于高斯。探索实际由 `entropy_coef=0.02` 控制（目前值 OK）。这行可以删掉避免混淆。

### 8. 奖励里的 `rew_smooth` 对离散动作不合理
`sum(actions^2)` 其中 `actions` 是从 `_action_map` 取出的 velocity 向量 `(±0.03, 0)` 或 `(0, ±0.03)`。所有非-stay 动作的 `sum(actions^2)=0.0009`，stay 是 0 — 这反而变相鼓励 stay，和 `rew_idle` 方向冲突。
建议：删掉 `rew_smooth`，或用 `action_change` 之类 "相邻动作不同则惩罚" 的形式。

### 9. reset 时 pitch fudge 系数 0.7
```python
# target_aiming_env.py:240-241
camera_height = self.cfg.tiled_camera_cfg.offset.pos[2]  # 2.0
aim_pitch = torch.atan2(torch.tensor(camera_height, device=...), dist_xy) * 0.7
```
用 `save_obs_images.py` 直接肉眼验证：如果 reset 后 car 普遍不在投影的绿圈附近，说明这个 fudge 不对，reward 的"真值"从一开始就是错的。

### 几何对齐（2026-04-19 追加，对应 Problem 9）
用户运行 `save_obs_images.py` 后发现 car 不在投影绿圈里。根因是 **投影代码硬编码 arm=2.0m，但 `tiled_camera_cfg.offset.pos` 被改成了 `(0.1, 0, 0.3)`**，projection_geometry.md 的校准前提失效。

选择了"短摆臂 + 近目标"路径（offset=(0.1, 0, 0.3)，target x 范围保持 1.0-2.0m）：

- `target_aiming_env.py::_project_target_to_image`：不再硬编码 `2.0/0.1`，改为 `self.cfg.tiled_camera_cfg.offset.pos` 读取 arm 长度；未来再改 offset 不会出现 reward/渲染不一致。
- `target_aiming_env.py::_reset_idx` 的 `aim_pitch` 计算：删除 `* 0.7` fudge（那是为 2.0m 长摆臂校准的补偿），改成精确的右三角：
  `atan2(pitch_link_z + offset.pos[2] - target_z, dist_xy)`。短摆臂下摆动效应 ≪ 1m，无需补偿。

### Policy collapse 修复（2026-04-19 追加）
首次 256×256 训练 ~200 iter 后崩溃：episode_length 从 300→25，mean_reward 从 ~2000→~75，且崩溃后曲线**完全平直无噪声**。Loss 曲线给出完整链路：
- `Loss/value` 在 iter 200 处从 ~5e3 暴涨到 1.7e4
- `Loss/learning_rate` 被 adaptive KL 折半连锁砍到 ≈ 0
- `Loss/entropy` 从 1.5（接近 ln5=1.61 的最大熵）瞬间归 0
- `Loss/surrogate` 同步归 0（已无更新）

根因：上一轮把 `rew_scale_success=10`、`rew_scale_pixel_center=1.0` 设得太大，单步 reward 上限 ~11，完整成功 episode 累计 ~3000，失败 episode ~5——600:1 的 returns bimodal 让 value 函数无法稳定拟合，一次大的 value loss 触发 KL 暴增 → adaptive lr 归零 → 探索消失，永久卡死。

修复（4 处联动）：

| 文件 | 字段 | 旧 | 新 | 作用 |
|---|---|---|---|---|
| `target_aiming_env_cfg.py` | `rew_scale_pixel_center` | 1.0 | 0.5 | 单步上限 11→1.6 |
| `target_aiming_env_cfg.py` | `rew_scale_success` | 10.0 | 1.0 | 消除 bimodal returns |
| `agents/rsl_rl_ppo_cfg.py` | `entropy_coef` | 0.02 | 0.05 | 熵下界更高，崩溃后能恢复 |
| `agents/rsl_rl_ppo_cfg.py` | `desired_kl` | 0.01 | 0.02 | adaptive lr 不那么激进 |
